# Linux Log Files Reference

## Overview

Linux operating systems generate log files that record authentication events, system activity, hardware messages, application logs, and administrative actions. These logs are essential for troubleshooting, security monitoring, digital forensics, and incident response.

This project primarily analyzes authentication logs while referencing other important Linux log files commonly used by system administrators and Security Operations Centers (SOCs).

---

# Linux Log Directory

Most Linux log files are stored in:

```text
/var/log
```

View the directory:

```bash
cd /var/log
ls
```

---

# Authentication Log (auth.log)

## Location

```text
/var/log/auth.log
```

## Purpose

The authentication log records security-related events involving user authentication and privilege escalation.

Examples include:

- SSH login attempts
- Successful logins
- Failed login attempts
- Invalid usernames
- Password authentication
- Sudo commands
- User session creation
- User session termination

Example:

```text
Jul 15 13:45:21 ubuntu sshd[2214]: Accepted password for aisha from 192.168.29.101 port 49152 ssh2

Jul 15 13:46:18 ubuntu sshd[2218]: Failed password for invalid user hacker from 192.168.29.105 port 49820 ssh2
```

Useful commands:

```bash
grep "Accepted password" /var/log/auth.log

grep "Failed password" /var/log/auth.log

grep "sudo" /var/log/auth.log
```

# System Log (syslog)

## Location

```text
/var/log/syslog
```

## Purpose

The system log stores messages generated by Linux services and applications.

Common events include:

- Network activity
- Service startup
- Service shutdown
- Cron execution
- System errors
- Hardware notifications

View the log:

```bash
sudo less /var/log/syslog
```

Search:

```bash
grep CRON /var/log/syslog
```

---

# Kernel Log (kern.log)

## Location

```text
/var/log/kern.log
```

## Purpose

Records messages generated by the Linux kernel.

Typical entries include:

- Driver loading
- Hardware detection
- USB devices
- Filesystem events
- Boot messages

View:

```bash
sudo less /var/log/kern.log
```

---

# Package Manager Log (dpkg.log)

## Location

```text
/var/log/dpkg.log
```

## Purpose

Records software installation and removal events.

Useful for determining:

- Installed packages
- Removed packages
- Updated software
- Package installation dates

Example:

```bash
cat /var/log/dpkg.log
```

---

# Journal Logs

## Location

Managed by systemd.

Access using:

```bash
journalctl
```

## Purpose

Stores centralized logs generated by:

- Kernel
- System services
- Applications
- Boot process
- Authentication events

Useful commands:

```bash
journalctl

journalctl -xe

journalctl -b
```

---

# Boot Log

## Location

```text
/var/log/boot.log
```

## Purpose

Stores messages generated during system startup.

Useful when troubleshooting:

- Boot failures
- Startup services
- Hardware initialization

---

# Alternatives Log

## Location

```text
/var/log/alternatives.log
```

## Purpose

Tracks updates made by the Debian alternatives system.

Useful when changing:

- Default editor
- Java version
- Python interpreter
- Compiler selection

---

# Log Rotation

Linux prevents log files from growing indefinitely through **logrotate**.

Check the configuration:

```bash
cat /etc/logrotate.conf
```

List rotation rules:

```bash
ls /etc/logrotate.d
```

Log rotation:

- Compresses old logs
- Removes outdated logs
- Creates new log files
- Saves disk space

# Security Events Monitored

This project focuses on detecting the following authentication events:

- Successful SSH logins
- Failed SSH login attempts
- Invalid username attempts
- Sudo command execution
- User session creation
- User session termination
- Multiple failed logins from a single IP address
- Potential brute-force attacks

# Summary

Understanding Linux log files is fundamental for system administration and cybersecurity. Authentication logs provide valuable information about user access and potential security incidents, while system and kernel logs help identify operational issues. By combining these log sources with Bash scripting, administrators can automate routine monitoring tasks, improve incident detection, and support digital forensic investigations.
